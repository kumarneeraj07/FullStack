# MongoDB to PostgreSQL/Sequelize migration

The entire backend data layer has been swapped from MongoDB/Mongoose to PostgreSQL/Sequelize while preserving the REST API contract for the React frontend. Models are rewritten as Sequelize classes with UUID primary keys, the booking concurrency mechanism moves from Mongo's single-document atomic updates to PostgreSQL row-level locking and a dedicated `SeatLock` table with a unique constraint, and every controller adds `_id` aliasing so the frontend never notices the change. The verify script now runs against SQLite in-memory instead of `mongodb-memory-server`, covering 61 integration tests.

**Watch for:** `sequelize.sync()` in production creates/alters tables without migrations â€” schema drift risk on deploy (confirmed). Password hash is no longer excluded from default queries, making it available on `req.user` throughout the request lifecycle (confirmed, not exposed in responses but expands attack surface). The `bookedSeats` column stored as a PostgreSQL `ARRAY(STRING)` creates a read-modify-write race window between concurrent confirmations even with `SELECT FOR UPDATE` (likely).

## High-level view

The concurrency model is redesigned around a `SeatLock` table with a unique constraint on `(showId, seat)`. Lock attempts INSERT into this table inside a transaction; the constraint catches duplicates, which map cleanly to 409 responses. Confirmation uses `SELECT FOR UPDATE` on the lock rows and the Show row to prevent interleaved updates. The `bookedSeats` column stores the full array and updates it via read-modify-write â€” the row lock serializes access, but the pattern is brittle: if a future code path touches `bookedSeats` without acquiring the lock, writes will be lost silently.

Every controller applies `_id` aliasing via local helper functions rather than a centralized serialization layer. Each controller's version is slightly different (some recurse into arrays, some don't), which means any new endpoint must remember to apply the transformation manually and match the correct variant.

Schema management relies on `sequelize.sync()` at startup. In production PostgreSQL this cannot remove columns, rename them, or alter types â€” the schema will drift from code over time. A migration tool is expected before production use.

The `User` model no longer hides the password hash from default queries. `User.findByPk` returns the full row including `password`, which is then attached to `req.user` for the lifetime of every authenticated request. No API response serializes it today, but the exposure window is wider than the Mongoose equivalent.

The SQLite/PostgreSQL dual-target strategy means tests never exercise PostgreSQL-specific behavior: row-level locking, `SELECT FOR UPDATE`, ARRAY operators, or true parallel transactions. The concurrency test validates the unique constraint sequentially, not under load.

## Booking concurrency via SeatLock table and row-level locking

The new design factors locks into a separate `seat_locks` table with a unique constraint on `(showId, seat)`, enforced at the storage engine level. The `confirmBooking` path acquires `SELECT FOR UPDATE` on both the user's SeatLock rows and the Show row before reading `bookedSeats`:

```
BEGIN
  DELETE expired locks
  SELECT FROM seat_locks WHERE ... FOR UPDATE  -- user's locks
  SELECT FROM shows WHERE id = ? FOR UPDATE    -- the show row
  read bookedSeats, append seats, UPDATE shows
  DELETE seat_locks for confirmed seats
  INSERT booking
COMMIT
```

The risk is the read-modify-write on the `bookedSeats` ARRAY column. Because the Show row is locked with `FOR UPDATE`, concurrent transactions queue rather than interleave, so in practice the array won't lose writes today. But the pattern is brittle: if a future code path reads `bookedSeats` without locking, it will overwrite a concurrent append. Using PostgreSQL's `array_cat` in a raw SQL `UPDATE` would make the append atomic regardless of locking discipline:

```sql
UPDATE shows SET "bookedSeats" = "bookedSeats" || ARRAY['A1','A2'] WHERE id = ?
```

This would also eliminate the need to SELECT the row before updating it, reducing lock hold time.

## `_id` aliasing inconsistencies

The `addIdAlias` helper is defined locally in each controller. The show controller's version recurses into arrays, the review controller's doesn't, and the booking controller's `myBookings` handler manually restructures nested associations with explicit destructuring. A centralized serialization utility would reduce the chance of a future endpoint forgetting to alias or aliasing inconsistently.

The `listShows` and `getShow` endpoints leave `bookedSeats` in the response (it's part of `...rest` after destructuring `Movie` and `Theatre`). This leaks the full list of booked seat IDs to any authenticated user who calls the show endpoint â€” a minor information disclosure that reveals how full a show is at the seat level.

## Schema management with `sequelize.sync()`

`connectDB` calls `sequelize.sync()` on every startup. In production, `sync()` runs `CREATE TABLE IF NOT EXISTS` plus `ALTER TABLE ADD COLUMN` for new fields. It will never:

- Remove a column that was deleted from the model
- Rename a column
- Change a column's type
- Drop a table

Over time, the production schema will drift from what the code expects. Use Sequelize migrations (via `sequelize-cli` or `umzug`) and disable `sync()` in production.

## Password hash exposure on `req.user`

The `authenticate` middleware calls `User.findByPk(payload.sub)` and assigns the result to `req.user`. Unlike Mongoose's `select: false`, Sequelize returns all columns by default. The hash sits on `req.user` for the lifetime of every authenticated request. If a future logging middleware, error handler, or debugging tool serializes `req.user`, the hash leaks.

Fix: add a `defaultScope` with `attributes: { exclude: ['password'] }` on the User model, and use `User.unscoped().findOne(...)` in the login path where the hash is needed for comparison.

## SQLite dual-targeting and test fidelity

The tests never exercise:

- `SELECT FOR UPDATE` (SQLite ignores it â€” only one writer at a time)
- PostgreSQL ARRAY operators (`Op.contains`, `array_append`)
- True concurrent transaction isolation (SQLite serializes all writes)
- `ENUM` type enforcement (SQLite stores ENUMs as plain strings)

The concurrency test locks a seat, then sequentially attempts to lock the same seat from another user. It validates the unique constraint but not the race condition that motivates `SELECT FOR UPDATE`. A proper concurrency test needs a real PostgreSQL instance and parallel HTTP requests.

## Expired lock accumulation

`releaseExpiredLocks` is called inside `lockSeats` and `confirmBooking`, but only for the specific `showId` being operated on. If a show has no new booking activity, its expired locks sit in `seat_locks` forever. They don't affect correctness (read paths filter by `expiresAt > now`) but accumulate storage and slow index scans.

Mitigation: a scheduled job that runs `DELETE FROM seat_locks WHERE "expiresAt" < NOW()` periodically, or a PostgreSQL index on `expiresAt` to keep the cleanup query fast when it does run.
